## Start here (read this first, do not re-derive it)

### Why Regimen exists (the main point; everything below serves this)

An engineer using an AI coding agent every day has a problem that is mostly about their own practice, not the model. When the work goes badly they often cannot say why; when it goes well they cannot reliably repeat what they did. There is no observability for the interaction between the engineer and the agent. **Regimen exists to close that gap.** The one question it is built to answer: *did the agent do what I wanted, or did I have to do a lot of correcting?* The only honest way to answer that is to record how the work actually went and be able to ask.

That answer-seeking is the **Feedback** pillar, and it is why the other two pillars exist in the order they do:

- **Guidance** instructs the agent through skills it is asked to follow. Advisory; the agent may or may not comply. Useful on its own. This is how you make a spotted problem go away once.
- **Enforcement** compels an outcome through any deterministic mechanism that takes the choice away from the model (hooks, permission and tool gating, pre-merge gates, sandboxing, schema-constrained output). Added when a behavior matters too much to leave to the model, for the iffy cases where Guidance alone is not reliable.
- **Feedback** observes how the work actually went and shows where the interaction is strong and weak. Added when you want to know whether Guidance and Enforcement are actually working, and what else needs sharpening. **Feedback is what makes the other two improvable.**

Regimen is **harness-agnostic and model-agnostic by design**: every schema, signal, interface, and config holds for any agent CLI and any model, and harness-specific detail is confined to a thin capture/adapter edge per harness. In the design's own words, adding a harness is meant to be "one new capture hook file plus one new translator." This push is the first real test of that promise across four harnesses at once.

### The headline outcome this push is for

The company has already landed on a fixed set of **evaluation questions** every engineer must answer about each harness. The payoff that makes this push worth doing: **feed those questions to an LLM that answers them grounded in the feedback Regimen recorded across all four harnesses.** That is the core "did this harness serve me?" question asked at evaluation scale, across weeks and across four CLIs, producing the deliverable the company actually needs instead of a from-memory guess at the end of the trial.

The user's stated bare minimum for this push: understand and execute the next steps to make the three-pillar installer work for all four harnesses, and stand up a minimal version of that cross-harness answering capability.

The transcript reader is not: each harness stores sessions in a different on-disk format, and `src/loader/rollout/codex-reader.ts` is a hardwired fold over Codex rollout JSONL with no abstraction, imported by name in `src/judged/assess.ts`. Every harness past Codex needs a net-new reader that produces the same v1 events plus judge-time content chunks from a different format, behind an interface that does not exist yet. This is real format-comprehension work per harness and it gates the judge for that harness, so it dominates the effort. Treat it as the first-class deliverable of each harness bring-up, not an afterthought.

**A.0 Harness-parametric refactor (do first; unblocks every harness).** Introduce a single harness descriptor/registry that maps each harness to: config home (env var plus default dir), hooks-config file path and format, the capture event-name set, the capture producer path, the skill target convention, and the session-resolution strategy. Then thread it through the six Codex-named seams so they take a harness rather than hardcoding it:
1. `resolveCodexHome` becomes `resolveHarnessHome(harness, ...)` in both feedback and enforcement; replace `--codex-home` with `--harness <name>` (home becomes derivable, with an explicit override).
2. `planCodexHooks` becomes `planCaptureHooks(harness, ...)`; `planGateHooks` takes a harness so events, producer command, and target file come from the descriptor.
3. Extract a `TranscriptReader` and `SessionResolver` port; make the Codex reader plus `src/codex/*` the Codex implementation behind it; change `src/judged/assess.ts` to resolve the reader/locator by harness instead of importing Codex by name. Generalize the `dataDir/codex/sessions` stamp path to `dataDir/<harness>/sessions`.
4. Gate command builders set `REGIMEN_HARNESS` from the descriptor, not the literal `codex`; fold the `rm-rf-gate-codex.ts` / `rm-rf-gate.ts` split into one parameterized gate.
5. `feedback install` and `installSkill` route through the descriptor; the hub forwards `--harness` instead of `--codex-home`.

This is a refactor of working, tested code, so it is TDD over an existing green suite, not a redesign. Use `architect-deep` only for the descriptor boundary itself.

OPEN DESIGN QUESTION (the FIRST thing the architect-deep pass must resolve, before any build): WHERE does the harness descriptor/registry live, and how do the repos consume it? Options: (a) a HUB-OWNED shared module that `regimen-feedback` and `regimen-enforcement` both import (one canonical source, but a real cross-repo dependency); (b) DUPLICATED per repo (no shared dep, but the capture edge and the enforcement edge can drift); (c) a PUBLISHED shared package (clean isolation, but adds release machinery). The descriptor is the central new artifact of the whole multi-harness effort and it spans all three repos (feedback threads it through capture wiring, the reader, and session resolution; enforcement through gate wiring; the hub forwards `--harness`), so this ownership decision is load-bearing and gates the refactor. Per ADR-0004 (repo boundaries) and ADR-0009 (the hub owns program-level cross-cutting concerns), a hub-owned descriptor is the likely answer, but architect-deep should weigh it against the cross-repo-coupling cost first. The REPO SEQUENCING for the refactor (which repo's change lands first) falls out of this decision and is the explicit first confirmation gate with the user.

**A.0 RESOLVED DESIGN (2026-06-18; architect-deep plus three read-only code scouts over feedback, enforcement, and the hub. Design only, no builds, awaiting the confirmation gate.)**

DECISION 1, descriptor ownership (the load-bearing first question): per-repo descriptors, NOT a hub-owned shared module and NOT a published package. The scouts overturned the earlier "hub-owned is likely" guess on the ADRs' own terms. The hub is a pure subprocess orchestrator that imports zero instrument internals and forwards `--harness <name>` as an opaque string, so it consumes no descriptor and has nothing to own and import; ADR-0009 explicitly forbids a dependency edge from the program root onto an instrument, which rules out option (a). There is no workspace, monorepo, or publish setup across the four repos (all Bun, all private, fully independent), so option (c) would add real release machinery and couple the repos against ADR-0004's independence principle. Feedback and Enforcement ALREADY duplicate the `Harness` enum, `asHarness`, `traceIdFor`, and the data-dir resolver on purpose, reconciled by the published `store-write-contract.md`. The descriptor's shared surface is the same kind of small frozen data, so it follows the same proven pattern: each repo owns its descriptor, and a new published "harness descriptor contract" freezes the shared subset both repos must reproduce identically.

DECISION 2, the data/behavior split (this is what makes Decision 1 clean rather than a compromise). The "descriptor" is two materially different things with two different owners. First, a pure-DATA descriptor of static facts: config-home env var plus default dir, the shared hooks-file path and its nested format, the skills subdir, plus the role-specific event set, producer command, and leaf marker. This is reproducible as frozen data in a contract doc and in both repos, and it is exactly the cross-repo shared surface. Second, feedback-private BEHAVIOR: the transcript reader, the session resolver, and the translator, which are functions that never cross a repo boundary and that Enforcement never needs. The in-repo module boundary (data vs behavior) lines up exactly with the cross-repo ownership boundary (shared vs feedback-private). That alignment is why the descriptor must stay pure data with behavior resolved separately: a descriptor carrying function refs could not be reproduced as frozen contract data across repos, so the split is FORCED by Decision 1, not just a cleanliness preference. (This is the one point where the four interface designs diverged. Two put the reader and resolver as function refs on a single descriptor object for a minimal single-lookup interface; the data/behavior split wins here only because the cross-repo ownership decision demands a data-shaped descriptor. In a single-repo world the minimal one-object design would have been the call.)

The three-layer model that results: (1) a shared data contract (harness enum, config home, shared hooks-file path plus format, skills subdir) as a new published contract doc, the analog of `store-write-contract.md`, reproduced as data in each repo; (2) per-repo descriptor data (Feedback: capture events, capture producer command, capture leaf marker; Enforcement: gate event, gate command builders, the `REGIMEN_HARNESS` value) as repo-private data on each repo's descriptor; (3) per-repo behavior registries (Feedback: a reader port and a resolver port keyed by harness) as repo-private functions, separate from the data descriptor.

DECISION 3, the two feedback-side seams (both clear the two-adapter bar, so both are real seams, not hypothetical). The `TranscriptReader` is one method, `read(content, {complete}) -> RolloutReadResult`, category 1 (pure string to structured, no I/O). `RolloutReadResult` is already harness-agnostic, so the seam is nearly free to cut and the ~650-line Codex fold becomes the Codex adapter in place with no logic change; two real adapters (Codex now, Claude next in A.1); tested by feeding fixture content directly, and the existing Codex reader tests retarget at the port verbatim. The `SessionResolver` is `resolveCurrent(ctx) -> sessionId | null` plus `locate({sessionsDir, sessionId}) -> {path, open} | null`, category 2 (filesystem, tested against a temp dir), with two STRUCTURALLY different adapters: Codex runs stamp then newest-scan then filename-regex, Claude takes the session id directly with no stamp and no scan. The strategy varies, not just the constants, which is exactly why this is a port and not parameterized constants. All four interface designs independently folded the "is this the live/open transcript?" decision INTO the resolver (the `open` flag on `locate`'s result), removing assess's current `newestRolloutName` plus basename coupling; adopt it, so assess forwards `complete: !open` and never learns how openness is decided. The composition root (`assess.ts`) keeps the `readFileSync`, so the seams never do I/O; assess takes a `harness` name plus an optional `sessionId` (resolved by the descriptor when omitted), does one `harnessSupport(harness)` lookup, and the entire fail-closed, anchor-insert, quarantine, and judge tail stays harness-blind and unchanged. The registry is one `harnessSupport(harness) -> {descriptor, reader, resolver} | undefined` composer, mirroring the existing `TRANSLATORS` map and living beside it; the existing translator registry stays as-is (it is on the loader/capture path, already harness-parametric and tested, so do not churn it).

DECISION 4, what NOT to build (seam discipline, one adapter equals a hypothetical seam). No `CaptureStrategy` union and no capture-as-tailer arm: today there is exactly one capture shape (the Codex hook-merge planner), and the Gemini-as-tailer path is only a "maybe" pending the version-pin finding, so capture stays descriptor DATA consumed by the existing planner; if Gemini later forces a tailer, that is a future seam with a real second adapter. Also rejected as one-adapter dressing: a `ConfigHomeResolver` port (the variance is two strings, already descriptor data) and a `CaptureHookPlanner` port (only Codex has a hooks planner today).

CONFIRMED REPO SEQUENCING (user-approved 2026-06-18, feedback then enforcement then hub). (1) Feedback repo FIRST, the deep, load-bearing one: introduce the data descriptor plus the reader and resolver ports plus the `harnessSupport` registry, thread it through `assess.ts`, the install wiring, and session resolution, and lift the Codex code into the Codex adapters in place. Feedback is first because it carries the riskiest surface (the reader and resolver ports, the gating risk this doc calls out) and the fullest descriptor, so proving the shape here makes Enforcement's smaller copy a trim rather than a guess, and Claude A.1 builds directly on it; the shared-data contract doc is drafted ALONGSIDE this work (the shared subset becomes clear as the descriptor takes shape), producer before consumer. (2) Enforcement repo SECOND: introduce its smaller data descriptor reproducing the frozen contract, fold `rm-rf-gate-codex.ts` and `rm-rf-gate.ts` into one parameterized gate that reads `REGIMEN_HARNESS` (the shell gates already do this), and parametrize `planGateHooks`, the gate command builders, and `resolveCodexHome` into `resolveHarnessHome`. (3) Hub THIRD: replace the `--codex-home` forward with `--harness <name>` in the locator, plan, and orchestration, with the hub still importing no instrument internals. There is no hard cross-repo build dependency (option b has no shared module to publish first), so this order is about de-risking, not compilation; each repo's change is independently green and mergeable.

CONFIRMED BRANCH/MERGE CADENCE (user-approved 2026-06-18; carried from the Codex push). A feature branch per repo per step (for example `a0-harness-descriptor` in feedback and enforcement, `a0-harness-forward` in the hub), in the relevant repo; this driver doc stays a living working-tree file, not committed unless asked. TDD over the existing green suite (real red-green-refactor, stub-before-red): this is a refactor, so the existing tests stay green and new tests cover the registry, the two ports, and the unknown-harness error, with the per-unit done-contract being `bun run check` green plus the unit's new tests. The CodeRabbit review gate runs on the EXACT merged commits, re-run after any amend or rebase; squash-merge only on explicit go; never auto-merge or auto-push. Each build unit routes off-thread to one delegated agent with the machine-checkable done-contract, while architect-deep (this) and the merge gate stay in the main thread.

DECISION 5, monorepo-readiness (new, from the user 2026-06-18). The user is leaning toward collapsing the multi-repo layout (the split that put feedback, enforcement, and the OTLP bridge in their own repos with `regimen` as a thin hub) into a single repo where those become subdirectories or workspace packages, because the split has created more complexity than it bought; the OTLP bridge was originally separated to keep it orthogonal to this effort, and that separation no longer feels worth its cost. Recommendation, accepted unless the user overrides: do A.0 NOW in the current multi-repo layout but build it CONSOLIDATION-FRIENDLY, and run the monorepo consolidation as its OWN architect-deep plus ADR effort AFTER the readiness work, not before it. Why after and not first: (a) the readiness work has a real external schedule (the work-VDI forensics this week, the Mac install next week, the four-harness trial), and a full repo restructure is a yak-shave that would block it; (b) A.0 is what REVEALS the real shared surface (the descriptor's data subset), so consolidating after A.0 means consolidating around a known shape rather than guessing it, which is producer-before-consumer applied to repo structure; (c) the consolidation is mechanical and low-risk once A.0 isolates the shared surface, so doing it first buys nothing; (d) the monorepo supersedes ADR-0004 (multi-repo by instrument) and touches the install front door, the clone-root resolution, and `bun link`, so it deserves its own deliberate pass rather than a rushed prelude.

The consolidation-friendliness CONSTRAINTS A.0 holds to (so the later monorepo is a delete-and-import, not a rewrite): (1) the shared-data subset lives in ONE clearly-marked module per repo (for example `harness-contract.ts`), authored to be BYTE-IDENTICAL across feedback and enforcement, so consolidation is "move one copy to a shared workspace package, delete the other, repoint two imports," and the hub's contract doc is what keeps the two copies identical until then; (2) the descriptor stays pure DATA (Decision 2), trivially extractable to a shared package, and the data/behavior split survives the monorepo unchanged (a shared data package imported by both, with the reader and resolver behavior staying feedback-private and never imported by enforcement, mono or multi); (3) NO new cross-repo build or release machinery (option c, a published package, stays rejected and is now doubly wrong given the monorepo intent), so there is nothing to unwind; (4) do not deepen the hub's subprocess-only coupling in ways that fight a future in-process model, the hub stays subprocess-only for now (it must, multi-repo) and A.0 does not make a later in-process switch harder. Net: this design is already robust to the structure change, because the only part of A.0 that was coupled to repo structure was the ownership decision, and the data/behavior split makes that decision a one-line import swap later.

RESOLVED SUB-QUESTION (user 2026-06-18): the shared-data contract doc lives in the hub's `docs/`, the analog of `store-write-contract.md` but program-level (ADR-0009 makes the hub the owner of program-level cross-cutting concerns, and the shared hooks-file path and format is precisely what the installer reconciles). Reframed by DECISION 5: the doc is both the shared-surface spec AND the explicit extraction target for the monorepo; once consolidated, its "reproduce identically in both repos" clause collapses into a single shared import and the doc becomes ordinary package documentation.

## Workstream B: Depth - the cross-harness evaluation-answering capability (the headline, minimal version)

This is the payoff. It is genuinely new: today's `feedback assess` is one LLM judgment over **one** conversation, written to a judged-signal schema and read back per conversation (every judged read is `WHERE session_id = ?`). Answering a fixed question set across a corpus is a different shape. The schema does not block it (harness is recoverable from `conversations` by join; ADR-0008 deliberately keeps harness and model off the judged row as read-time slice dimensions), so this is an aggregation read path plus a new answering pass, not a migration. Minimal build:

**B.1 Read-time harness/model slice on the judged store.** Join judged signals and narratives back to `conversations.harness` (and scoped events for model) so reads can group and compare by harness. Wiring only; closest existing issue is #23, whose acceptance already says "sliced by model and harness."

**B.2 A corpus-selection primitive.** Choose which conversations feed a given answer: by harness, date window, intent, repo/cwd. Does not exist today; presupposed by #23/#22.

**B.3 The question-to-answer pass (new orchestration, distinct from per-conversation assess).** Feed the company's evaluation questions plus the selected recorded feedback (judged signals, narratives, and the evidence behind them) to the LLM and get back answers grounded and anchored in real sessions. Recommended shape for comparability: answer each question **per harness** (grounded in that harness's corpus) plus a cross-harness summary, matching the PRD's "comparable across harnesses." Record provenance (which runs and sessions fed each answer) so an answer stays falsifiable, the same anchoring discipline the per-conversation judge already uses. Closest issues: #10 (rollup-verdict structure) and #24 (respond-step surfacing), under epic #4.

**B.4 A surface to read the answers.** Minimal first: a CLI command that prints the per-harness answers. The richer human browse CLI (#15) and the time-series bridge (#32) are deferred.

**Dependency:** B is only as good as its corpus. "Across all four harnesses" requires Workstream A capture to exist for those harnesses (regimen-feedback #4 is the gating coverage dependency). Until A.1 to A.3 land, B answers over whatever harnesses are captured (Codex now, Codex plus Claude after A.1).
