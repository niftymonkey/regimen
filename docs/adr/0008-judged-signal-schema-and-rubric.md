# The judge writes typed signals through one generic anchored row, narrative as separate prose, and the in-session read leads with the conversation assessment

> Status: accepted (2026-06-14)

The judgment layer answers "did this harness serve me?" for one conversation by writing typed, evidence-anchored signals plus a prose assessment, and the in-session agent reads them back as a verdict, not as restructured evidence counts. This ADR settles what the judge outputs and how it scores: the storage shape every judged signal takes, how a signal references the captured events that justify it, how prose lives apart from scored values, how a re-judge supersedes a prior one, and the wire contract that `feedback assess` (#21) writes and the in-session judge skill (#25) reads. It fully specifies the two signals the assess spine emits, Intent and Outcome, and reserves the other signals `docs/feedback-surfacing.md` enumerates as a named roster whose shapes are decided when they are built. It designs the output and its anchors only: it builds none of the judge prompt, the transcript content projection (deferred to the reader by ADR-0007), the assignment segmenter, or the in-session reader.

## Why this is its own decision

The Purpose of the trial-readiness effort is to use Regimen the real way, which is the judgment layer: Feedback that returns a judged verdict on a conversation with the evidence behind it, surfaced to the agent in the tight loop. The evidence layer (capture, store, deterministic signal projections, the in-session evidence read) is built and is facts-only by ADR-0005; interpretation belongs to the judge. So the judge needs a place to put interpretation, and that place is a new contract, not an extension of the evidence digest. `docs/feedback-surfacing.md` enumerates the judge's signals and the principle that a count or rate describes activity, not how the work went, so the judged output must be tighter than the evidence read or the two layers collapse into one. Everything downstream of this design (assess in #21, the in-session skill in #25, and the stretch long-arc work in #22, #23, #24) writes or reads against the contract decided here, which is why it is settled before any of them is built.

## One generic signal row, not a column per signal

The recurring cost in this layer is the signal roster itself: `docs/feedback-surfacing.md` already lists nine per-assignment signals and says more follow. A column or table per named signal makes each addition a schema migration and fixes every signal's shape cold, before the judge has run once against a real transcript. So a judged signal is one generic row, not a named column: `judged_signal { session_id, scope, assignment_id, signal_name, value_kind, value, run_id }`. `signal_name` is a closed controlled vocabulary, which is the guard that keeps a future harness adapter from smuggling a harness-shaped signal into the judged store. `value_kind` is a small, open type tag (`categorical` and `ordinal` are the live kinds; further kinds arrive with the signals that need them when #22 builds them) and `value` is JSON typed by that tag, not an untyped escape hatch. The headline signals stay sliceable because `signal_name` plus a normalized `value` make them queryable exactly like a counted signal would be. Adding any of the deferred signals is then data within the existing shape, not a migration. Model and harness are never denormalized onto the row; they are read-time slice dimensions resolved from the events within the signal's scope, not from the session-level `conversations` rollup, because a conversation can span models and the doc requires that a single model be assessable on its own terms. Resolving from the scoped events attributes each assignment's signals to the model that actually produced that work, keeps model out of the judged row, and means a swap to a neutral judge model or a new harness needs no change here; the session-level `conversations.model` stays a convenience for the common single-model case, never the authority for judged-signal attribution.

## Scope, the assignment reservation, and the reserved roster

A signal's `scope` is `conversation` or `assignment`, and it carries a nullable `assignment_id`. The assignment is the bounded unit of work within a conversation, and `docs/feedback-surfacing.md` divides a conversation into one or more assignments through segmentation. The segmenter is stretch work (#22), so the assess spine writes exactly one whole-conversation assignment, `assignment { session_id, assignment_id }`, and every per-assignment signal foreign-keys to it. Reserving the nullable `assignment_id` now is the cheap forward move: when #22 lands, segmentation populates an existing reference instead of re-keying rows that cannot be rebuilt. The reservation is all that is built now. The segmenter's own output fields (segment boundary anchors, an ordering index) have no meaning for a single conversation-spanning assignment and are designed when #22 builds them.

This ADR fully specifies only the two signals the assess spine emits. Intent is `categorical`: one value from a closed, harness- and model-neutral vocabulary naming what the engineer was trying to do (`refactor`, `bug-fix`, `feature`, `test-writing`, `exploration`, `schema-change`, and an explicit `other` so a wrong fit is never forced), one per assignment, and also the dimension other signals group by at read time. Outcome is `ordinal`: one value from a rank-ordered set, `abandoned` below `partial` below `accomplished-with-correction` below `accomplished-cleanly`, scored on task accomplishment and the amount of steering it took, and explicitly not on transcript length and not on software quality. The rank order is load-bearing, because trending and comparison read it as a spectrum. The other seven signals `docs/feedback-surfacing.md` names (correction rate, correction types, prompt clarity, drift, struggle, expressed dissatisfaction, silent acceptance) are a reserved roster: their names are recorded, and each signal's `value_kind`, vocabulary, cross-checks, and absence semantics are decided when #22 builds it against real transcripts. One forward constraint binds that later work: a signal's deterministic operand is never copied into the judged row. Correction rate, for example, resolves its prompt-count denominator from the evidence layer at read time, so the judged store holds only the judged numerator and the evidence layer stays the single source of the deterministic count.

## Signals carry anchor references into the events store, never copied content

An anchor, in the domain language, is the deterministic event from the evidence layer that the judgment layer points at when making a claim. A judged signal does not hold the event; it holds a reference to it, because a claim must be falsifiable (ADR-0003 forbids the opaque verdict) and conversation content lives in the transcript and never in our store (ADR-0005). So the wire sub-shape is an anchor reference, not the event and not free text: `AnchorRef = { eventHash } | { sessionId, toolCallId }`, where `eventHash` is the lowercase-hex encoding of the `events.event_hash` value. That column is a BLOB primary key (a raw sha256), which does not round-trip through the JSON wire contract, so the encoding is named here as the thing `feedback assess` and the in-session reader must share; without it the reference silently fails to resolve back to a row. A signal that cites a moment and every narrative row carries `anchors: AnchorRef[]` with at least one entry, and a claim may carry several. Reusing the keys ADR-0006 already guarantees means a re-render produces the same references for free and introduces no new id space.

## Narrative is separate prose with no scored value

`docs/feedback-surfacing.md` keeps the narrative outputs apart from the signal store, so narrative is a physically separate table, `narrative { session_id, scope, assignment_id, narrative_type, prose, anchors, run_id }`, with no typed value and no part in any rollup arithmetic. `narrative_type` is an open text discriminator; the assess spine writes exactly one type, `assessment`, the per-conversation reasoning, and later phases add other prose types as data. The discriminator is open text rather than a frozen set, because an open column is the whole forward affordance and enumerating future prose types would design the stretch respond-step surfaces (#24) into this foundation. The assessment prose is generated before the Outcome value, so reasoning precedes the label and the typed signals stay lean while the prose and its anchors carry the show-your-work burden. There is no per-signal free-text rationale field, which would duplicate the narrative and bloat the trended store.

## Re-judging supersedes through a run identity

Re-running the judge over a conversation must not duplicate its signals, which is an acceptance criterion of `feedback assess` (#21), so a judgment pass has an identity: `assessment_run { run_id, session_id, rubric_version, prompt_version, judge_model, complete, created_at }`. Every signal and narrative row carries its `run_id`, and the latest run wins, superseding per `(session_id, scope, assignment_id, signal_name)`. Stamping `rubric_version` and `prompt_version` on the run is what makes a re-judge after a rubric change detectable rather than a silent overwrite. `judge_model` records which model did the judging as opaque provenance, mirroring the nullable model in ADR-0006; nothing in the rubric, the type tags, or any read-time query may branch on its contents, because the moment code switches on it the judged layer acquires a model-specific behavior. `complete` means the run finished rather than aborted; per-assignment abstention is expressed by the signal simply being absent, because absence is first-class throughout this design. No judged row carries its own `schema_version`: schema growth is additive through the single shared `schema_migrations` ledger ADR-0006 mandates.

## The output contract

`feedback assess` (#21) writes, in one pass over one conversation, exactly one `assessment_run`, one whole-conversation `assignment`, the judged signals (Intent and Outcome at minimum), and one `assessment` narrative. A missing transcript exits with a clear error and never a half-judged result; insufficient evidence on the conversation is recorded as the run being incomplete and the signal absent, never as a fabricated verdict. The in-session judge skill (#25) reads a `JudgmentDigest`, the judged twin of the evidence layer's `EvidenceDigest` and a separate contract from it, so the two layers stay two surfaces. It is a discriminated union on a `judged` boolean. The unjudged branch is a genuinely different shape that mirrors `EvidenceDigest`'s unknown branch and is produced by its own constructor, never the judged shape with empty placeholders; it covers Feedback being off, no run yet, or the transcript being gone. The judged branch leads with the conversation assessment, the `scope=conversation` narrative and its anchors, because the assessment is the conversation-scoped artifact and stays defined whether the conversation holds one assignment or many. Outcome stays per assignment in the drill-down: for the single whole-conversation assignment the assess spine writes, that lone Outcome sits directly beneath the assessment, so the in-session agent reads "did this serve me?" without reconstructing it. A conversation-level Outcome is never stored or promised, because Outcome is per-assignment by the doc and a conversation aggregate would be a signal the doc does not name; any single-assignment headline convenience is the reader's to derive, not a contract field. The assignments and their signals are the drill-down beneath the headline. The reader sorts deterministically so a re-judge of the same input is stable; the exact tie-break is the reader's call when it is built. The reserved-roster signals are stored when they exist but are not in the default read projection, because a bag of typed values surfaced to the agent would be the evidence digest wearing a judged label.

## Considered options

- **A column or table per named signal, instead of one generic row.** Rejected. The roster grows, so a column per signal makes every addition a migration and fixes nine signal shapes before the judge runs once. The generic row keeps Intent and Outcome queryable today and lets the rest arrive as data, which is the only shape consistent with a roster `docs/feedback-surfacing.md` already says will grow.
- **A flat read contract where the agent reads Outcome out of a sorted signals array by name.** Rejected for the in-session surface. It is purer, but it makes the mid-session agent reconstruct the headline and gives the judged read the same value-bag shape as the evidence read, which collapses the two-layer distinction the judgment layer exists to create. The judged branch leads with the verdict instead.
- **Specifying all nine signals' shapes, vocabularies, and cross-checks now.** Rejected. The assess spine emits Intent and Outcome; specifying the other seven is designing the #22 signal set under this ADR and locking vocabularies cold before they are tuned against real transcripts, which is the over-build the generic row exists to avoid. They are reserved by name and built where they are used.
- **A numeric composite quality score on top of Outcome.** Rejected. Fine numeric scales are where an LLM judge goes arbitrary and unactionable, the four-value Outcome ordinal is the headline, and ADR-0003 forbids grading software quality.
- **A top-level verdict field, or a third middle state between judged and unjudged.** Rejected. The verdict is concretely the Outcome plus the assessment, not a new stored concept, and the binary judged-or-not union plus a note already expresses fail-closed-on-unknown; partialness is carried by the run's `complete` flag and by per-signal absence.
- **A per-row `schema_version`, an open attributes map on judged output, or a per-signal rationale field.** Rejected. Schema growth is the shared ledger's job per ADR-0006, the judge emits a controlled vocabulary so it needs no escape hatch, and a per-signal rationale duplicates the narrative.

## Consequences

The judged store is four tables (`assessment_run`, `assignment`, `judged_signal`, `narrative`) on the shared migrations ledger, with one shared `AnchorRef` sub-shape referencing the existing events keys and one `JudgmentDigest` wire contract that `feedback assess` writes and the in-session skill reads. The generic row means the seven reserved signals are added as data when #22 builds them, against real transcripts, each with its own vocabulary and cross-checks, under the forward constraint that no deterministic operand is copied into a judged row. The nullable `assignment_id` reservation lets segmentation (#22) populate an existing reference rather than re-key rows. The headline-led read means the in-session agent (#25) gets an actionable verdict without reconstructing it, and the deferred signals stay out of the default projection until they have an assignment to mean something. The transcript content the judge reasons over is read through the one per-harness reader ADR-0007 names, not designed here. This ADR resolves the design pass tracked as issue #8, and is the foundation issues #21 and #25 build against next.
