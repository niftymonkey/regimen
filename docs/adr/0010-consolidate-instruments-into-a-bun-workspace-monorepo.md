# Consolidate the instrument repos into one Bun-workspace monorepo; the instrument boundaries survive as packages

> Status: accepted (2026-06-19)

ADR-0004 cut Regimen into separate repos per instrument so each could be installed and deployed independently, and judged a monorepo as something that would fight that independence. That call has not held up in practice. The split bought independent installability we never exercise (every instrument is installed together by one orchestrator on one machine) and charged a recurring tax we pay constantly: the same small facts are duplicated by hand across `regimen-feedback`, `regimen-enforcement`, and `regimen-otlp-bridge` and kept in sync only by discipline and a contract doc, the four repos carry four byte-identical `tsconfig.json` files and four `bun.lock` files, and the conversations that drive this work run in the `regimen` clone while the code being changed lives in a sibling clone, so the conversation cwd is never where the work is. This ADR supersedes ADR-0004's physical decision: fold `regimen-feedback`, `regimen-enforcement`, and `regimen-otlp-bridge` into the `regimen` repository as Bun workspace packages under `packages/`. It changes only the physical layout. The logical instrument boundaries ADR-0004 drew survive intact, now as package boundaries rather than repo boundaries.

The timing is deliberate, not opportunistic. The A.0 harness-parametric refactor (ADR-0007 readers, the harness contract, the data/behavior split) is built and merged, so the cross-instrument shared surface is now a known shape (`src/harness/contract.ts`, the `Harness` set, `traceIdFor`, the data-dir resolver) rather than a guess. Consolidating now means consolidating around that known surface, producer-before-consumer applied to repo structure. It also lands before the macOS trial install, so the trial validates the structure Regimen will keep instead of a multi-repo install front door that is about to be replaced.

## The layout: every instrument, including the CLI, is a package under packages/

The repository root becomes a pure workspace container: a private `package.json` declaring `workspaces: ["packages/*"]`, the shared dev toolchain, one `bun.lock`, one base `tsconfig` that each package extends, the relocated Husky pre-commit hook, and the program-level documents that belong to no single instrument (the ADRs, the PRD, ARCHITECTURE, DOMAIN-LANGUAGE, the plans). It owns no instrument source.

Five packages live under `packages/`, with bare directory names and `@regimen/*` package names now that the repository scope makes the old per-repo `regimen-` prefix redundant: `cli` (`@regimen/cli`, the orchestrator and the `regimen` bin, the package formerly called the hub), `feedback`, `enforcement`, `otlp-bridge`, and one new shared-data package, `shared` (see below). The name change is not cosmetic: once the program-level documents move to the root, the former hub is no longer a container of docs and issues, it is purely the command-line entry point and the subprocess orchestrator, so it is named for what it is. The CLI package moves into `packages/cli` rather than staying at the root, and that choice is load-bearing for the install front door. It resolves each instrument's clone root through a three-level precedence in `src/locator.ts`: an explicit path flag, then an environment override, then a named-sibling convention that joins the parent of the CLI package's own root with the instrument's conventional directory name. The flag and environment legs are layout-independent and do not change. The convention leg's resolution logic is preserved unchanged precisely because the CLI becomes a sibling of the other packages: with it at `packages/cli`, the parent of its own root resolves to `packages/`, and the same join lands on `packages/feedback`. Dropping the directory prefix does change the convention's directory constants (from `regimen-feedback` and `regimen-enforcement` to `feedback` and `enforcement`) and the locator test's fixture directory names, but those are string substitutions the tests catch immediately, not a change to the resolution logic. The runner already routes each spawned subprocess with its cwd set to the resolved clone root, so `bun link` and every instrument invocation keep acting on the right package directory. The single highest-risk line in the whole consolidation, the join that the trial install exercises and that fails late, therefore keeps its logic intact under this layout. Had the CLI stayed at the root, that join itself would have needed reworking to insert a `packages/` segment, the avoidable structural risk this layout removes; demoting the change to a pair of renamed string constants is the point.

## The shared-data package is a pure leaf, and the data/behavior split survives the move

The one genuine refactor in the consolidation is extracting the duplicated shared surface into a package (`@regimen/shared`, at `packages/shared`) that both Feedback and Enforcement import, with the bridge as a third consumer of its data-dir resolver. Applying the deletion test: if this package never existed, complexity does not vanish, it concentrates across three consumers plus a contract doc whose only job is to keep three hand-copied copies identical. So it earns its keep. What concentrates behind it is small and entirely category-1, pure computation with no I/O: the known-harness set and its narrowing helper (`Harness`, `asHarness`), the OTLP trace-id derivation (`traceIdFor`), the data-directory resolver (`resolveDataDir` over an injected env and platform, plus the `dataDir`/`bufferDir` location helpers the store-write contract freezes), and the harness contract data (`src/harness/contract.ts`, already authored to merge with zero diff). Because every member is pure computation, the package exposes no port and injects no adapter. It is a data-and-helpers leaf, imported directly.

This is the data/behavior split from the A.0 design surviving the structure change verbatim. The shared package holds data and pure helpers only. The behavior that reads transcripts and resolves sessions, the `TranscriptReader` and `SessionResolver` ports and their per-harness adapters, stays Feedback-private: it does filesystem I/O, it varies structurally per harness, and Enforcement never needs it, mono-repo or multi. Enforcement's consumption shrinks to imports: the `Harness`/`asHarness`/`traceIdFor`/`resolveDataDir` it currently inlines in `denial-store.ts` come from the shared package, while its own `RegimenEvent` and gate-denial row shapes stay local (see the next section). The bridge keeps its private path helpers (the feedback-db, watermark, and bridge-log paths it builds on top of the resolved data dir) and its store-row projection types, which mirror nothing in Feedback's schema; only its data-dir resolver folds into the shared import.

What deliberately does not go into the shared package is as important as what does. There is no shared event-envelope type, no shared store-row schema, and no shared eslint or prettier config package. The store-row shapes are governed by a contract, not a type (next section). The toolchain config is covered by a single root base `tsconfig` that packages extend and root-level lint and format config, the native workspace mechanism, so a published config package would be dead ceremony.

## The store-write seam stays a contract, not a shared type

Feedback publishes its store as an open format: a documented store-write contract plus a conformance test, the seam ADR-0005 and ADR-0009 lean on, across which Enforcement's denial emitter writes without importing Feedback's code. The consolidation keeps that seam exactly as it is. Its value is independence from the language and the repo: any future producer, a non-TypeScript hook, a different harness's native gate, an external tool, can write into the store by conforming to the documented row shape, and that value does not come from a shared module that only TypeScript callers can import. So the de-duplication is scoped to helper code, not to the wire format. Both producers go on reproducing the `RegimenEvent` row shape against the published contract; the shared package gives them the `traceIdFor` and data-dir helpers the contract references, and the conformance test's inline re-derivation of the trace id stays as a drift guard, now guarding the shared package's `traceIdFor` against the written contract. Collapsing the row shape into a shared type would quietly close the open-format seam to everything that is not TypeScript, which is the opposite of what the seam is for.

## Git history: clean-cut, with the originals archived read-only

Bring each instrument in as a working-tree copy and record the consolidation as a small set of ordinary commits in the `regimen` repository, rather than stitching each repo's history under its package directory with `git subtree`. The repos are young and single-author (the four source repos carry on the order of tens of commits each, not thousands), so the history being interleaved buys little, and subtree's merge commits and braided log would complicate `git log`, `git bisect`, and blame in the consolidated repo for that small gain. Pre-consolidation history is not lost: the original repos are kept as archived, read-only sources of truth, so any blame or log archaeology for a change made before the move is one clone away. The one consequence to accept plainly is that `git blame` inside the monorepo dates each moved file to the consolidation commit, not to its original authorship; for code this young, with the originals preserved, that is an acceptable trade.

## What this ADR does not change

This is a physical-layout decision and is deliberately narrow. It does not move the unified installer or alter ADR-0009: the installer stays in Feedback until ADR-0009's own trigger fires, and its end-state home is still the orchestrator (ADR-0009's "hub," now the `@regimen/cli` package). One wording note follows from the move: ADR-0009 phrases part of its trigger as "a standalone `regimen-enforcement` repo with its own install verb." Read that, post-consolidation, as "a standalone `@regimen/enforcement` package with its own install verb." The intent, two or more independently installable components each behind its own stable install verb, is unchanged; only the noun moves from repo to package. It does not change ADR-0005's open-format store seam, ADR-0007's per-harness readers, or the A.0 data/behavior split, all of which carry over intact. It does not touch the orchestrator's subprocess-only coupling to the instruments; it still spawns each instrument's CLI as a child process and imports zero instrument internals, so a future in-process model is neither helped nor foreclosed here. And it leaves the `skills` content repo named in ADR-0004 out of scope: that is curated Guidance content installed into the cross-tool skills directory, not a TypeScript package, and it has no place in this workspace.

## Considered options

- **Keep multi-repo by instrument (ADR-0004's decision).** Rejected. The independence it protects is never exercised (one orchestrator installs every instrument together on one machine), while the duplication tax, the fourfold toolchain and lockfiles, and the cwd-versus-code mismatch are paid continuously. The cost is real and the benefit is hypothetical.
- **Monorepo with the CLI left at the repository root and the siblings under `packages/`.** Rejected in favor of moving the CLI under `packages/` too. Leaving it at the root forces the convention-leg join itself to change (to insert a `packages/` segment), which is an edit to the exact resolution logic the trial install runs and that fails late, and it conflates the workspace-container role with the CLI-package role in one `package.json`. Making the CLI a peer package preserves the sibling-resolution logic unchanged (only the directory-name constants are renamed) and keeps the root a clean container.
- **`git subtree` to preserve each repo's history under its package directory.** Rejected. For repos this young and single-author the preserved history is low-value, and the braided log and merge commits raise the ongoing cost of `git log`, `bisect`, and blame. Clean-cut plus archived originals keeps the history reachable without paying that cost in the live repo.
- **Extract the store-row and event types into the shared package alongside the helpers.** Rejected. It would close the open-format store seam to non-TypeScript producers, defeating the reason the seam exists. The seam stays a documented contract; only helper code de-duplicates.
- **A published shared package, or a shared config package.** Rejected. A published package would add release machinery and is doubly wrong given the monorepo intent; the shared surface is consumed in-workspace. Toolchain config is handled by a root base `tsconfig` and root lint and format config, the native workspace mechanism, so a config package would be ceremony with no payoff.
