# The unified `regimen` CLI dispatches to instruments in-process; the harness-invoked scripts stay standalone

> Status: accepted (2026-06-22)

ADR-0009 settled that the unified installer is a program-level concern owned by the `regimen` entry, and described the shape that orchestrator must take when the trigger fired: "a thin shell-out orchestrator, not an import-based module that reaches into any instrument's interior. It invokes each instrument's own install verb as a subprocess and owns only program-level concerns." ADR-0010 then folded the instrument repos into one Bun-workspace monorepo and explicitly preserved that posture, leaving "the orchestrator's subprocess-only coupling to the instruments" untouched so a future in-process model was "neither helped nor foreclosed." This ADR fires that future. It supersedes the "shape of the move" section of ADR-0009 only: the `regimen` CLI dispatches to `@regimen/feedback` and `@regimen/enforcement` as in-process libraries rather than spawning their CLIs as subprocesses, the locator and runner and plan modules that bridged the process boundary are deleted, and the per-instrument `feedback` and `enforcement` bins collapse into the single `regimen` bin. Everything else in ADR-0009 is reinforced, not changed: the installer is still a program-level concern, and each instrument's install logic still lives in its own package.

## Why the subprocess posture no longer earns its keep

ADR-0009's case for subprocess-only was a repo-boundary case. With Feedback, Enforcement, and the bridge in separate repos with their own secrets and deployment, an import-based installer would have reached across a closed boundary and recreated the bundling ADR-0004 forbade, so spawning each instrument's CLI and forwarding exit codes was the correct shape. ADR-0010 dissolved those boundaries. The instruments are now workspace packages that already import each other in-process: Enforcement imports `@regimen/shared` for the harness contract, the trace-id helper, and the data-dir resolver. With the boundary gone, the justification for crossing it by subprocess is gone with it, and the locator, runner, and plan modules become a multi-repo fossil: they exist only to find a sibling clone on disk, spawn its bin, and marshal flags into argv, none of which an in-process call needs.

Applying the deletion test to each confirms the collapse rather than a relocation. The locator's entire depth is resolving a sibling instrument's clone root so a subprocess can run its bin; an in-process caller resolves the package by workspace import, so that complexity vanishes. The runner's one real seam was an injected spawn function with a real adapter and a recording-fake adapter, and that two-adapter seam existed only because there was a subprocess to fake; in-process there is nothing to fake, so the seam vanishes and the tests move from spawn-and-assert-exit-code to call-and-assert-result. The plan module turned an install config into argv-bearing steps and routed flags to the right child; in-process a caller passes a typed options object to each library function, so the argv construction and flag routing vanish and only the ordering invariant survives, as call order plus a test rather than as a data structure. Three modules and their tests delete; the design introduces no new port in their place, because every remaining dependency is in-process computation or a filesystem seam already substituted by a temp directory in tests.

## The shape of the in-process orchestrator

The `regimen` entry becomes the one composition root and the one argv parser. It resolves the harness once, dispatches each subcommand to a typed library function exported by the owning package, and holds a single error and exit contract. It stays deliberately shallow: the depth lives in the library functions it calls and in the new lifecycle module, not in the dispatcher.

Each instrument package exposes its commands as callable library functions that take a parsed options object and return a result, rather than only a `runCli(argv)` that parses argv internally. This is the load-bearing refactor. The alternative considered, keeping each `runCli(argv)` and having the dispatcher rebuild an argv string to call it, was rejected: it marshals already-parsed options back into strings for a sibling to re-parse, keeps three argv parsers alive, and can return only an exit code, which is the deleted subprocess seam wearing an in-process disguise. Typed command functions make each command's interface the surface its tests exercise directly.

What the orchestrator owns does not otherwise change from ADR-0009. The per-instrument install logic, the daemon planner, the skill planner, and the hooks merge stay local to the package whose artifacts they install. The orchestrator composes them and owns the genuinely cross-instrument concerns: the ordering that lands the capture hook ahead of the gate on a pre-tool boundary, the cross-instrument reconciliation of one harness hooks file, and now the install lifecycle below.

## The harness-invoked scripts stay standalone, and that is the safety property

The collapse touches only the user-facing command layer. Every script a harness invokes during a session is referenced by absolute path in installed configuration, never through the `feedback` or `enforcement` bin: the capture hooks the harness runs on each event, the gate scripts and the denial emitter the harness runs on a pre-tool boundary, and the loader daemon the operating system supervisor runs from the recorded loader path. These stay standalone entrypoint scripts, invoked by path exactly as before. Dropping the per-instrument bins therefore cannot break an installed machine, because nothing installed ever pointed at those bins. This is what makes the consolidation safe to ship between a working install and the next, and it is what makes an in-place update tractable: the only thing an upgrade must rewrite is the set of absolute paths baked into the installed hooks, gates, and service definition, which is precisely what the lifecycle module re-resolves.

## The command surface drops the pillar nouns

The unified surface is grouped by user task, not by instrument. The pillar names that ADR-0001 and ADR-0002 cut for the internal architecture, Feedback and Enforcement, leave the command line entirely, because a user judging a session should not have to know which instrument owns the verb. Program lifecycle verbs are flat and unambiguous: `regimen install`, `regimen update`, `regimen uninstall`, `regimen status`. The read and judge primitives that are the daily verbs are flat: `regimen assess`, `regimen evidence`, `regimen list`. The daemon supervisor is grouped to avoid colliding a daemon `status` and `start` with the program-level verbs: `regimen daemon start|stop|restart|status`. The wiring verbs that used to be top-level, wire-hooks, wire-gates, install-daemon, and install-skill, become internal steps of `regimen install` rather than user verbs, so the install grows deeper and the surface grows smaller.

## The install lifecycle: a manifest makes the install knowable

The orchestrator gains a lifecycle that the multi-repo installer never had, because the question the macOS and Windows acceptance runs raise, "what is installed, at what version, for which harnesses, and in which workspaces," has no answer today. Install state today is implicit and scattered: an enabled-flag file, a daemon pid file, the existence of a service definition, and the Regimen markers stamped into each harness hooks file. None of it records a version, and there is no update verb, so the only safe upgrade today is to uninstall and reinstall.

The lifecycle is one new deep module behind a small interface, persisting an install manifest at the resolved data directory. Applying the deletion test, the manifest concentrates real complexity rather than passing it through: without it, version, update, and install-record logic would smear across the install command, an update command, and every ad-hoc reader of the data directory. It hides the manifest persistence format and the update algorithm. The manifest records the Regimen version read from the CLI package metadata as the version stamp, the install and update timestamps, the resolved clone path and loader path, and one entry per installed harness carrying its pillars and its install scope. The scope is load-bearing: Gemini capture and gates install per workspace under ADR-0011, so an entry must record whether it landed in the harness config home or in a named workspace, or an update cannot re-install Gemini where it actually lives.

`regimen update` reads the manifest, re-resolves the clone path and loader path from the current clone so a moved or upgraded clone rewrites the absolute paths baked into the installed hooks, gates, and service definition, re-runs the idempotent per-harness install for each recorded entry, cycles the daemon so it runs the new loader path, and restamps the version and update time. With no manifest present it falls back to behaving like a fresh install, since it cannot know what to update. `regimen install --all` and `regimen install --harnesses` loop the per-harness install, with Gemini remaining per workspace and a one-line notice to that effect. The module depends only on the filesystem at the data directory, substituted by a temp directory in tests, and the version value read from package metadata, so it needs no port.

The dependency category is the resolution of the open question the multi-repo era left for the work VDI: with no version stamp and no update verb, the safe upgrade is uninstall-first, because a bare reinstall only adds or replaces and never removes. Once this lifecycle lands, an in-place `regimen update` is safe, because the uninstall path cleans the prior artifacts (the stable identifiers, the scheduled task name, the data directory location, the hook markers, and the skill names are byte-identical across versions) and the data directory survives so capture history is preserved.

## What this ADR does not change

It does not touch ADR-0009's core decision that the unified installer is a program-level concern owned by the `regimen` entry, nor that per-instrument install logic stays local to each package; both are reinforced. It does not change ADR-0010's package layout, the `@regimen/shared` leaf, or the data and behavior split. It does not change ADR-0005's open-format store-write seam: Enforcement's denial emitter continues to write across the documented contract, and the consolidation of the user-facing CLI does not make the store a shared in-process type. It does not change ADR-0011's per-workspace Gemini install; it records that scope in the manifest so the lifecycle can honor it. And it does not alter any harness-invoked script: the capture hooks, gate scripts, denial emitter, and loader daemon keep their standalone entrypoints and their absolute-path invocation.

## Considered options

- **Keep the subprocess orchestrator (ADR-0009's shape).** Rejected now that ADR-0010 removed the repo boundary that justified it. Spawning a sibling package's bin to do work the caller can do by a typed function call retains a locator, a runner, a spawn seam, and an argv round-trip whose only remaining purpose is to cross a boundary that no longer exists. The subprocess posture was correct under multi-repo and is dead weight under the monorepo.
- **Dispatch in-process but keep each package's `runCli(argv)` and rebuild argv to call it.** Rejected. It avoids the facade refactor but marshals parsed options back into argv strings for a sibling to re-parse, keeps three argv parsers, and can return only an exit code. It is the deleted subprocess seam in an in-process disguise, and it forfeits the testability the typed facade buys.
- **Keep the per-instrument `feedback` and `enforcement` bins alongside the unified `regimen` bin.** Rejected. The bins exist for users and skills to invoke commands directly, which the unified CLI now owns; keeping them entrenches the very surface this ADR removes and leaves the pillar nouns on the command line. The harness-invoked scripts that must stay standalone are separate entrypoint files referenced by absolute path, not these bins, so dropping the bins costs nothing installed.
- **Defer the install lifecycle until a per-OS run reveals concrete update needs.** Rejected for the manifest and version stamp, which are the prerequisite that turns the work VDI's upgrade question from uninstall-first into in-place update; building the stamp now is what makes the per-OS acceptance runs a turnkey execution rather than a fresh investigation. The richer update ergonomics can still grow after the first real runs without reopening this decision.
