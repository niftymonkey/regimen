# The unified installer is a hub concern; it ships from Feedback for the trial and moves to the hub when a second instrument self-installs

> Status: accepted (2026-06-15)

The unified installer stands up full Regimen on a machine: the Feedback daemon and capture hook, the Guidance skills, the Enforcement gates, and eventually the optional OTLP bridge and any further component. That is a program-scoped, cross-cutting concern that belongs to no single instrument, so its end-state home is the hub (`regimen`), not the Feedback repo. This ADR records that decision now, while the installer physically lives in `regimen-feedback` as a deliberate trial expedience, so the placement is settled rather than relitigated later and so `feedback install` is understood as a temporary name, not the answer. It also fixes the trigger that moves the installer to the hub and the shape that move must take. It moves no code now: the tested `regimen-feedback` installer ships as-is for the Codex trial.

Update: realized differently than sketched here. The unified installer now lives in the hub as the `@regimen/cli` package and dispatches to each instrument's install logic in-process within one monorepo (ADR-0012, ADR-0010), not as a shell-out across separate repos. The "three pillars" framing in this ADR is superseded by ADR-0013, which makes Feedback the center with Guidance and Enforcement the levers.

## Why the installer is not a Feedback feature

ADR-0004 cuts repos by instrument and names `regimen` (hub), `regimen-feedback`, `regimen-enforcement`, and `skills`, with the OTLP bridge separate. It is silent on where a concern that spans every instrument belongs, and the installer is exactly that concern. The evidence is on disk: the bootstrap describes itself as standing up all three pillars; the hooks merge writes both Feedback's capture hook and Enforcement's gate commands into one `~/.codex/hooks.json`; the gates record denials into Feedback's store through Feedback's own data dir. So Feedback and Enforcement are already coupled at two seams (the shared hooks file and the shared store), and the installer is the thing that reconciles them. Only one node legitimately depends on every instrument at once, which is the hub. Homing the orchestrator permanently in Feedback inverts that dependency arrow, and once the bridge is added it recreates precisely the bundling ADR-0004 forbade, Feedback reaching across to a sibling that has its own secrets and deployment. The forward requirement that the installer optionally install the OTLP bridge and other components confirms this: its terminal dependency set spans repos that do not depend on each other, which is a hub shape, not a Feedback shape.

## The seam test is already tripped; trial sequencing is why the move waits

The honest reason the installer stays in Feedback for now is sequencing, not that the trigger is unmet. Two built, real instruments (Feedback and Enforcement) are already wired by this one orchestrator, and the bridge exists as a third repo, so the one-adapter-versus-two-adapter test that governs extraction is genuinely satisfied today. The accident that Enforcement currently lives inside `regimen-feedback` (its gates and the denial emitter) is expedience debt to be corrected, not evidence that Enforcement is not its own instrument; ADR-0004 already names its repo. That debt must not be used to argue the installer should stay in Feedback.

What does defer the move is cost against a deadline. The installer is not thin glue today: the Feedback CLI imports several Feedback internals in-process (the store, the evidence read, the judged assess path, the enabled flag, session resolution), and most of those serve Feedback's other subcommands, not the install. Lifting the orchestrator to the hub now would force either a dependency edge from the program root onto one instrument, or designing the hub-to-instrument install seam against a single self-installing consumer while the bridge's own install path is still unbuilt. Designing that seam for an audience of one is the premature generalization the seam test warns against, and it would mean standing up a greenfield hub package and re-greening the installer's tests mid-trial. So the decision is to commit the hub as the end-state in this ADR and ship the tested Feedback branch for the trial.

## Per-instrument install logic stays local; the hub owns only the thin composition

The piece that belongs in the hub is the composition, not the per-instrument install logic. The daemon planner, the skill planner, and the Codex hooks merge are correctly local to the repo whose artifacts they install, and they should stay there. The hub owns only the thin layer that composes each instrument's own install verb plus the genuinely cross-instrument concern, reconciling one `~/.codex/hooks.json` across instruments. Conflating the two (lifting the whole installer, including Feedback's in-process internals, into the hub) is the trap, because it drags Feedback's interior across a repo boundary that should stay closed.

Refinement: the hub reconciles `~/.codex/hooks.json` by sequencing two idempotent role-scoped planners (Feedback's `role:"capture"` planner, then Enforcement's `role:"gate"` planner, each preserving foreign leaves), not by owning a reconciler module. The intent is preserved; the mechanism is sharper.

## The trigger to extract

Move the composition layer to the hub the moment the first of these lands, whichever comes first: a standalone `regimen-enforcement` repo with its own install verb, so the gate command specs leave Feedback and the hub composes Feedback and Enforcement across declared install interfaces rather than in-repo clone paths; or the bridge's planned `bridge install-daemon`, giving the second self-installing component the forward requirement already names; or any third instrument gaining its own install verb. The unifying condition is that the installer must compose two or more independently deployable components, each behind its own stable install verb. Today exactly one such verb exists (`feedback install`, which also wires the co-located Enforcement gates). Because cutting `regimen-enforcement` is itself within our control, the trigger is partly a choice, not only an external event: the decision here is to not force it before the trial, and to not let it slip indefinitely either, since every month the orchestrator stays `feedback install` deepens both the wrong name (the program wants a `regimen` verb that adds, removes, and updates individual instruments) and the wrong dependency direction.

## The shape of the move

When the trigger fires, the hub installer is a thin shell-out orchestrator, not an import-based module that reaches into any instrument's interior. It invokes each instrument's own install verb as a subprocess (`feedback install`, then `enforcement install`, optionally `bridge install-daemon`) and owns only program-level concerns: the cross-instrument `~/.codex/hooks.json` reconciliation, instrument discovery (replacing the clone-rooted path resolution with named per-instrument inputs), and ordering. Feedback's in-process internals do not move; they stay with the subcommands that use them. Extracting `regimen-enforcement` is part of the same motion: the gate scripts and the denial emitter leave Feedback, and the gate command set becomes install metadata the hub consumes. One prerequisite gates that cut: the Enforcement-to-store write path (the denial emitter appending into Feedback's store) must first be published as a documented store-write contract over the ADR-0005 open-format seam, so that once Enforcement is its own repo it writes across that seam rather than through an in-process import. To keep the future lift cheap, the hooks merge already takes its gate set as opaque data, so the Enforcement payload can be lifted out wholesale without reworking the merge.

## What is built now versus deferred

Now: this ADR, and shipping the tested `regimen-feedback` installer for the trial unchanged. Nothing moves. Deferred to the trigger: standing up the hub as a code repo, the thin shell-out orchestrator and the hub-owned hooks reconciler, the `regimen-enforcement` cut, and the published store-write contract that must precede it.

## Considered options

- **Keep the installer in Feedback permanently.** Rejected as the steady state. It is the right place to build the per-instrument logic, and the right place to ship from for the trial, but making it the permanent home entrenches a name that under-describes a three-pillar action and, once the bridge is added, the cross-to-a-sibling dependency direction ADR-0004 forbids. It also leans on the Enforcement-in-Feedback expedience debt, which is exactly what this ADR says must be corrected.
- **Lift the whole installer to the hub now.** Rejected on timing, not on placement. The orchestrator is not yet thin glue, so moving now forces a program-root-onto-instrument dependency edge or a hub-to-instrument install seam designed against a single consumer while the bridge's installer is unbuilt, plus a greenfield hub package and a mid-trial test re-green. Correct destination, wrong moment.
- **Merge the Feedback installer for the trial and defer the whole placement question.** Rejected. Shipping from Feedback for the trial is right, but deferring the decision risks relitigating it and lets the `feedback install` name and dependency direction entrench by default. Recording the hub as the end-state now makes the eventual move mechanical.
