# Each harness has one transcript reader that owns its format; it recognizes records by shape and fails closed on the unknown

> Status: accepted (2026-06-14)

A harness's on-disk session transcript is read by one per-harness transcript reader: a pure fold over the transcript content that emits the harness-agnostic v1 structural events the evidence layer already consumes. There is exactly one reader per harness, because it is the single place that harness's transcript format lives. This ADR settles the one piece ADR-0005 deferred to a future ADR: how that reader handles a transcript format the vendor evolves and does not treat as stable. Rather than fingerprinting the whole transcript to a version, the reader recognizes each record by its shape: it parses the record shapes it knows, fails closed on a known load-bearing record whose fields it cannot parse, and skips but counts and surfaces a record type it has never seen, so format drift is always visible and never a silent misread. The reader's content projection for the judge is out of scope here and is settled when the judgment layer is designed; the only commitment this ADR makes about it is that it reuses this same reader rather than a second one that re-learns the format.

## Why this is its own decision

ADR-0005 records that the judge reads conversation content from each harness's transcript via a per-harness transcript reader, and explicitly defers the per-reader versioning design to a future ADR. This is that ADR. The transcript is a vendor artifact Regimen does not control, and Codex states outright that its session-file format is not a stable interface. Today the Codex rollout reader is hard-pinned to the single build shape (codex-cli 0.128.0) it was tested against, with no version negotiation, and it silently skips any record it does not recognize. Without a versioning design, the next Codex release that changes the shape silently breaks the reader, and nothing signals that it broke. The reader is the seam where vendor change meets the normalized v1 events the rest of Regimen depends on, so the seam is where the versioning discipline lives.

## One reader per harness owns the format

The expensive, recurring complexity here is vendor-format knowledge: which record in the transcript means what, in which generation of the format. That knowledge must live in exactly one place per harness, or every vendor change becomes a multi-site edit and the copies drift, which is the precise failure this ADR exists to prevent. So each harness has one transcript reader, and it is the sole owner of that harness's format. The reader is a deep, in-process module: a small interface (transcript content in, normalized events out) over a body that holds all the format and record handling. Parsing is pure computation and the file's location and read are an ordinary local filesystem concern, so the reader needs no injected port; it is tested by feeding it fixture transcripts and asserting on the events it returns, which is how the Codex rollout reader is already tested.

This single-owner rule is also what governs the judge. ADR-0005 has the judge read conversation content from the transcript; that content reading is deferred to the judgment-layer design and is not built here. The one commitment this ADR makes is that when it is built, the judge reads content through this same per-harness reader, not a second reader that re-learns the same format. Structural events for the evidence layer and content for the judge are two uses of one format-owner, so a vendor change stays a one-place fix.

## Recognizing format: per record, not per transcript

The unit of recognition is the record, not the transcript. Whole-transcript version detection is unsound for this kind of format: a transcript carries only a subset of the format's record types (the build-variance finding on Codex was that a 0.128.0 session that edits no files emits no patch records at all), so absence of a record type is ambiguous, it could mean an older format that lacks the feature or a session that simply never used it. A resumed or long-lived session can also outlast a vendor upgrade and mix shapes in one file. So the reader does not try to name the transcript's version; it recognizes each record by its shape and parses the ones it knows.

Most vendors do not stamp a format version anyway. Codex's session metadata carries a CLI version, but that is the version of the CLI, not of the format, and it is kept only as a hint about which record shapes to expect, never as the authority. The reader holds a handler per known record shape (a record type with a recognized field arrangement) and dispatches each record to its handler; the handlers live inside the one reader so a harness's known shapes stay in one place and the parsing they share is reused.

Fail-closed has two cases, split by whether the unrecognized record is load-bearing. A known load-bearing record (a session boundary, a prompt, a tool call or its result) whose fields do not match a shape the reader can parse is quarantined and surfaced through the CLI, never best-effort parsed, because misreading it would feed the evidence layer and the eventual judge wrong data. A record type the reader has never seen at all is skipped but counted and surfaced, so a benign new auxiliary record does not fail an otherwise-readable transcript, while the surfaced count makes any drift visible and is the signal to add a handler. The honest residual risk, stated plainly: a future build that renames a load-bearing record degrades to a visible drop in expected events plus a rising unknown-record count rather than a hard failure, so those surfaced counts, not a silent zero, are what a reader of the evidence must watch. This is the "Honest over tidy" principle from `docs/feedback-surfacing.md` applied to format drift: nothing the reader cannot parse is ever passed off as if it had been parsed.

## From a vendor format change to an updated reader

Because an unrecognized record is always visible (a quarantined load-bearing record, a rising unknown-record count, or the judge's explicit decline), a vendor format change is a bounded, observable event rather than a silent corruption. The change surfaces as a record shape the reader does not recognize, which is the trigger to add one record handler plus a fixture captured from the new shape, a single-file change with its own regression test, leaving every older shape still handled. A record either matches a handler and is parsed, or it does not and is reported; there is no middle state where it is parsed wrong.

## Relationship to the portability studies

The per-harness portability studies in `niftymonkey/skills` are Regimen's knowledge base for how each harness behaves, with a freshness model the reader's format knowledge should inherit rather than reinvent: a `Verified:` date per study, "Known volatile" and "Known stable" callouts, per-fact primary-source citations, and a re-verification recipe. Today those studies document runtime affordances (hooks, tools, memory files, permissions) and they capture each harness's hook input JSON shape, which is the capture-edge format the translators read. They do not yet document the on-disk transcript format the reader reads. This ADR records that gap: the transcript format becomes a study facet, a new section in each harness study or a sibling study under the same framework, so a reader's record handlers trace to a dated, re-verifiable description the same way the translators already trace to the documented hook-input shape. "This transcript format is explicitly not a stable interface" is exactly the kind of fact the studies' "Known volatile" callout exists to hold.

## Considered options

- **Fingerprint the whole transcript to a format version.** Rejected. A transcript carries only the record types its session happened to use, so absence of a type is ambiguous and cannot pin a version, and a resumed session can mix shapes in one file. Recognizing each record by its shape sidesteps all of this and degrades at the granularity of the actual change.
- **Detect the format version from a stamped marker (the CLI version) alone.** Rejected. The marker is the CLI's version, not the format's: it can stay constant while the format changes and change while the format holds. It is kept only as a hint about which record shapes to expect.
- **Silently skip any record the reader does not recognize (today's behavior), or best-effort parse it.** Rejected. Both feed the evidence layer and the eventual judge wrong or missing data with no signal, the exact failure ADR-0005 and "Honest over tidy" forbid. Unknown records are surfaced: a malformed load-bearing record is quarantined, an unknown record type is counted and reported.

## Consequences

The reader stays a deep in-process module tested with fixture transcripts, and no new port enters the design. The Codex rollout reader is the existing instance; giving it per-record recognition with the surfaced fail-closed behavior, in place of its current hard pin to one build and its silent skipping of unknown records, is implementation work tracked separately, not performed here. Supporting a new harness's transcript adds one reader alongside the capture hook and translator that ADR-0005 already names. A vendor format change becomes bounded and visible: an unrecognized record is surfaced, and the fix is one record handler plus a fixture. The portability studies gain a transcript-format facet so the reader's record handlers trace to dated study descriptions; adding that facet and backfilling the studied harnesses is a follow-up in `niftymonkey/skills`. The judge's content reading stays deferred to the judgment-layer design (issues #21 to #25 under epic #4), with the single constraint that it reuses this reader. This ADR resolves the design pass tracked as issue #14.
