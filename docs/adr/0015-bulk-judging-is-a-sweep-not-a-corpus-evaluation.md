# Bulk judging is a sweep of the per-conversation judge, not a corpus-level evaluation

> Status: accepted (2026-06-29)

Regimen captures the evidence layer for every conversation automatically, but the judgment layer (the LLM-as-judge, `assess`) is produced only by invoking it by hand, one conversation at a time. To evaluate how work has gone across many conversations, an engineer needs the judgment layer to exist for the whole corpus, not just whichever conversations they remembered to assess. This ADR adds that as `regimen assess --all`: a command that selects conversations through the existing `list` filters and runs the existing per-conversation `assess` on each, skipping those already judged unless `--force` is given. It is deliberately a loop over the unchanged per-conversation judge: no new storage, no per-conversation behavior change, and no aggregation layer. The cross-conversation answer ("how did they all go") stays compositional and out of band, as decided when the baked `answer` command was dropped (see ADR-0013, Feedback is the center).

## Why a sweep, not an aggregate judge
The expensive, irreducible unit is judging one conversation. Once judged, a conversation's verdict is persisted (`assessment_run` / `judged_signal` / `narrative`, keyed by session id) and reading it back is a free, deterministic query. So "across many conversations" decomposes cleanly: make each per-conversation verdict exist (the sweep), then read or roll up the persisted verdicts (a separate, free concern). Folding many conversations into a single judge call would invent a second species of judgment with its own rubric, schema, and failure modes, for no gain the loop does not already provide, and would couple corpus-reading to the metered path.

## Cost is metered, so spending is batched and observed, not estimated
The judge defaults to Opus 4.8 over a metered Anthropic key, and a full-corpus sweep can be many calls. Rather than estimate a cost the engineer cannot convert to credits in their head, the sweep runs in batches (default 10, `--batch N`) and pauses between them for a single keypress (continue one batch, run all remaining, or quit), so real provider usage can be checked against the credits actually spent before the next batch begins. The first batch always runs, because invoking the sweep is itself the opt-in to spend; a non-interactive run (piped or backgrounded, no tty) stops after that first batch instead of silently judging everything, so a sweep can never run up an unbounded bill unwatched. Skip-already-judged makes the sweep resumable for free: a quit, or that non-interactive stop, is resumed simply by issuing the command again. `--force` re-judges everything for when the judging mechanism itself changed, and the per-run answering-model id makes such a change detectable. The sweep is continue-on-error and self-narrating: each conversation prints a contiguously numbered progress line as it is judged, a conversation whose transcript has aged out is recorded as a failure and the sweep moves to the next, and an end-of-run summary names what was judged, what failed, and what was left unjudged, so the spend is watched as it happens rather than guessed at up front.

## What this does not change
`assess` without `--all` behaves exactly as today. There is no new storage, no new per-conversation logic, no aggregation layer, and no automatic trigger. Selection reuses the `list` filters verbatim. The transcript constraint stands: `assess` reads each conversation's transcript from the harness files at judge time, so the sweep judges only conversations whose transcripts survive.

## Considered options
- A new verb (`regimen sweep` / `judge-all`). Rejected: it invents a name for what is just `assess` over a wider target, splitting "run the judge" into two concepts the domain language does not have.
- A corpus-level judge reasoning over many conversations in one call. Rejected: a second judgment species with its own rubric and schema that couples the cross-conversation read to the metered path; the per-conversation verdict plus a free rollup yields the same answer.
- A stored aggregation/rollup built now. Deferred, not rejected: reading persisted verdicts is free and compositional, consistent with dropping the baked `answer` command; build it when the read need is concrete. The shape that read should take (two co-equal capabilities: a verdict rollup of the per-conversation assessments, and a time-scoped check of whether the engineer's own established practices are honored across conversations) is a separate decision, not this one.
- A dry-run cost estimate before spending. Rejected in favor of empirical batching: an estimate the engineer cannot map to credits is not actionable, whereas a first real batch is.
- A trivial-conversation pre-filter (skip very short conversations). Dropped during design as premature; addable later if short conversations prove a real cost.
